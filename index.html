各具特色的五大神器

幫你找出最適合自己開發風格的神隊友，並理解每一套工具最適合放在學習流程的哪個位置。

Antigravity

自主執行的無形特工

#Agentic #多步推理

深度融入環境的高階系統，具備完整自主性，不僅能寫 code，還能自己下指令、讀文件、跑測試，解決跨檔案系統性的問題。

  • ✔️ 高度自主的計畫與執行
  • ✔️ 可以直接與開發環境互動
  • ⚠️ 需要對複雜工具有一定理解
🎓 給學生的建議：
適合給專題生或研究生做架構翻新與深度開發體驗。

Claude Code

終端機裡的最強大腦

#CLI #Anthropic

無縫切入你的終端機 (Terminal)。具備 Claude 最新的推理模型優勢，能在純文字介面極速修改與理解你整個 repo 裡的程式碼。

  • ✔️ 指令列愛好者首選
  • ✔️ 速度極快且精準度極高
  • ⚠️ 沒有圖形化跨檔案視窗
🎓 給學生的建議：
資工系學生的最愛，習慣用 vim/nano 跟終端機的人必裝。

Codex

會讀 repo、會執行任務的程式代理

#Agent #OpenAI

Codex 類型工具強調「直接在程式專案裡工作」：能讀取多個檔案、依需求修改程式、執行測試並回報結果。比起單純聊天，更像能協助完成整段開發流程的 AI 助理。

  • ✔️ 適合做跨檔案重構與功能實作
  • ✔️ 會根據 repo 脈絡做較完整的修改
  • ⚠️ 仍需人類確認需求與輸出品質
🎓 給學生的建議：
很適合拿來做期末專題、作業修補與 code review 練習，但一定要搭配版本控制觀念一起使用。

Cursor

AI 原生圖形化 IDE

#IDE #開箱即用

直接改寫 VS Code 核心的 AI Native 編輯器。擁有超強的多行「Tab」預測以及 Command K 直接變更程式碼的神奇體驗。

  • ✔️ 學習曲線最平緩
  • ✔️ Cmd+K 直覺變更多個檔案
  • ⚠️ 進階功能需付費訂閱
🎓 給學生的建議：
想感受 AI 魔力的最佳新手村，完全沒有痛苦的轉換期。

GitHub Copilot

老牌且穩固的補齊王者

#學生免費 #廣泛整合

市佔率最高！直接作為擴充套件安裝在任何你習慣的 IDE。邊寫邊幫你補齊，讓你專注在邏輯而不是語法上。

不只比較，還要會用：大學生 AI 開發學習攻略

以下內容適合直接拿來當作課堂練習、自學筆記或專題開發的操作參考。

課堂講義版整理

建議教學定位：把 AI 工具視為「輔助學習與驗證的實驗平台」

若要把這些工具真正導入課堂，建議不要只示範「它能幫你寫多少」，而要引導學生比較：需求描述品質、程式正確性、解釋能力、測試意識與版本控制習慣。換句話說，評量重點不只是產出結果，而是學生是否理解 AI 生成內容的理由與限制。

#課堂示範 #專題實作 #AI素養

可直接納入課程的三個學習目標

學習目標 A：正確描述問題

學生能把模糊需求轉換成具體 prompt，包括輸入／輸出格式、限制條件、驗證方式與例外情境。

學習目標 B：評估 AI 產出品質

學生能判斷回覆是否正確、是否遺漏邊界條件、是否符合演算法與資料結構課程中的核心觀念。

學習目標 C：建立可重現的開發流程

學生能保留版本、紀錄修改原因、驗證測試結果，並知道何時應回退、重寫或拒絕採用 AI 建議。

Step 1：先定義任務，不要一開始就問「幫我寫完」

AI 最怕的是需求太空泛。建議先把題目拆成「輸入是什麼、輸出是什麼、限制條件是什麼、你目前卡在哪裡」四件事。這樣工具才不會亂猜，回覆也會更接近你真正要的內容。

  • 作業情境：請 AI 先解釋題意，再提供解法步驟。
  • 專題情境：先描述系統架構，再指定某一個模組修改。
  • 除錯情境：貼上錯誤訊息、預期結果與目前程式片段。

Step 2：依工具特性分工，效率會比只用一套更高

不同工具不一定互相取代，反而常常是互補。用 Copilot 做日常補齊、用 Cursor 做圖形化探索、用 Claude Code 或 Codex 做跨檔案任務，會比拿單一工具硬撐所有情境更有效率。

  • 快速補語法：GitHub Copilot
  • 一邊看檔案一邊問問題：Cursor
  • 終端機精準改檔：Claude Code
  • 跨檔案任務與實作流程：Codex / Agent 類工具
  • 長流程整合與研究整理：Antigravity 類工具

Step 3：一定要要求 AI 說明「原因」與「驗證方式」

只拿答案不問原因，很容易變成表面上寫完，實際上卻不會。建議每次都補問：為什麼這樣寫？有沒有替代方案？我要怎麼測試？這樣才真的能把工具變成老師，而不是作弊機器。

  • 請它解釋時間複雜度與空間複雜度。
  • 請它列出邊界條件與測試案例。
  • 請它比較兩種實作在可讀性與效能上的差異。

Step 4：Codex 類工具怎麼用才不會失控？

Codex 這類能直接操作 repo 的工具，強項是把需求轉成具體修改。但使用時一定要先定義範圍，例如「只改前端頁面」「只新增某個函式」「修改後要跑哪些測試」，這樣才能維持可控、可驗證、可回滾。

  • 先要求它閱讀現有檔案並整理修改計畫。
  • 再要求它分批修改，而不是一次大改整站。
  • 最後要求它列出 diff 重點與驗證結果。

建議評量規準：學生用了 AI，不代表可以少做思考

1. 問題定義

是否清楚界定任務範圍、輸入輸出、資料格式與成功條件。

2. 解法理解

是否能口頭或書面說明 AI 產生的演算法、流程與設計理由。

3. 驗證能力

是否提供測試案例、邊界條件、錯誤處理與輸出檢查方式。

4. 工程習慣

是否使用版本控制、保留修改紀錄，並能回溯每次改動的原因。

推薦工作流：從作業到專題都適用

1

理解題目

先請 AI 幫你把題目改寫成簡單語言，確認輸入、輸出、限制與評分標準。

2

規劃解法

先產生流程圖、演算法步驟或模組拆分，再決定要不要開始寫程式。

3

局部實作

從最小可驗證單位開始，例如一個函式、一個頁面、一個 API，而不是整包一次產生。

4

測試與修正

要求工具列出測試資料、例外情境與可能 bug，再逐步修補。

授課建議：不同課程可如何導入

資料結構

要求 AI 生成 linked list、stack、tree 的實作後，請學生比較時間複雜度、記憶體配置方式與程式可讀性。

機率與統計

把 AI 當作解題步驟講解員，要求它拆解推導流程，並由學生檢查公式前提與結論是否一致。

IoT / 嵌入式應用

讓 AI 協助產生 Raspberry Pi 範例程式，但要求學生自己驗證腳位、感測器資料格式與實測結果。

專題製作

將 Codex、Claude Code、Cursor 分工使用：一套看架構、一套做精準修改、一套支援 IDE 協作。

可直接套用的提問範例

概念學習型

「請用大一學生看得懂的方式解釋二元搜尋樹，並給我一個 C 語言小範例，最後再出 3 題練習題。」

除錯型

「這段 Python 在輸入空字串時會報錯，請先找出根因，再提供最小修改版本，最後列出 5 組測試資料。」

專題實作型

「請先閱讀這個前端專案的 HTML/CSS/JS，提出改版計畫，再分三步完成：新增區塊、改善文案、保持手機版可讀性。」

Codex / Agent 協作型

「請先分析 repo 結構，僅修改 index.html 與 style.css，新增教學內容區塊與工具比較項目；修改後請回報變更摘要與檢查結果。」
